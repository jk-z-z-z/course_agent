# 后端代码规范

本文档只描述后端工程实现规范，不承载任何具体业务模块说明。

## 1. 目标

这份规范用于约束：

- 目录层级
- 分层职责
- 单一职责边界
- 命名方式
- 错误处理
- 数据访问方式
- 测试和交付要求

目标是让不同开发者在同一套后端代码上协作时，结构一致、职责清楚、便于维护。

## 2. 总体原则

所有后端代码默认遵守以下原则：

- 单一职责
  - 一个文件、一个结构、一个函数应尽量只处理一类问题
- 分层明确
  - HTTP、业务、数据访问不能混写
- 依赖单向
  - handler 依赖 service，service 依赖 repository，repository 不反向依赖上层
- 显式输入输出
  - 不依赖隐式共享状态，不通过猜测字段含义完成逻辑
- 易于替换
  - 核心逻辑尽量与具体框架、数据库细节解耦
- 先可读，再技巧
  - 优先写容易继续维护的代码，而不是追求复杂抽象

## 3. 推荐目录结构

下面是推荐结构，可以按项目实际栈微调，但职责边界不能乱：

```text
cmd/
  server/
internal/
  bootstrap/
  router/
  handler/
  service/
  repository/
  model/
  dto/
  vo/
  middleware/
  response/
  errors/
  utils/
pkg/
configs/
scripts/
tests/
```

目录职责：

- `cmd/server`
  - 程序入口，只负责启动应用
- `internal/bootstrap`
  - 配置、数据库、缓存、日志、依赖注入初始化
- `internal/router`
  - 路由注册、中间件挂载、模块装配
- `internal/handler`
  - HTTP 请求处理、参数绑定、返回响应
- `internal/service`
  - 业务逻辑编排、规则校验、事务控制
- `internal/repository`
  - 数据库读写和查询封装
- `internal/model`
  - 数据库模型
- `internal/dto`
  - 请求入参结构
- `internal/vo`
  - 对外响应结构
- `internal/middleware`
  - 认证、日志、跨域、恢复等通用中间件
- `internal/response`
  - 成功和失败响应封装
- `internal/errors`
  - 业务错误定义与映射
- `internal/utils`
  - 无业务语义的通用工具

## 4. 分层职责

### 4.1 Handler 层

handler 只负责：

- 接收请求
- 绑定参数
- 读取上下文
- 调用 service
- 返回统一响应

handler 禁止做的事：

- 直接写 SQL 或直接操作 ORM
- 写核心业务逻辑
- 拼装复杂跨模块流程
- 直接返回不统一的 JSON 结构

### 4.2 Service 层

service 是业务逻辑唯一归属地，负责：

- 业务规则校验
- 多 repository 协作
- 事务边界控制
- 状态流转
- model / dto / vo 之间的转换编排

service 编写约束：

- 不依赖 HTTP 请求对象
- 不直接感知页面交互细节
- 一个函数只完成一个明确业务动作
- 复杂流程拆小函数，不在单函数中堆叠所有分支

### 4.3 Repository 层

repository 只负责数据访问：

- 查询
- 创建
- 更新
- 删除
- 条件过滤

repository 禁止做的事：

- 做权限判断
- 写 HTTP 语义
- 拼装页面返回格式
- 夹带与数据访问无关的业务逻辑

## 5. 单一职责规则

以下规则必须明确执行：

- 一个 handler 文件只处理一个资源或一个子模块
- 一个 service 文件只处理同一类业务动作
- 一个 repository 文件只封装同一类实体的数据访问
- 一个函数超过 60-80 行时，优先考虑拆分
- 一个函数同时出现参数校验、权限校验、事务、查询、格式化返回时，通常说明职责混杂

判断是否应拆分的简单标准：

- 是否同时处理“输入解析”和“业务决策”
- 是否同时处理“业务决策”和“数据落库细节”
- 是否同时处理“正常流程”和“大量错误映射”

如果答案是“是”，优先拆分。

## 6. DTO / VO / Model 规范

### 6.1 DTO

DTO 只用于接收输入参数：

- 必须声明清晰字段名
- 必须有序列化 tag
- 必须有校验 tag 或等价校验方式
- 更新接口优先使用指针字段区分“未传”和“传空值”

DTO 不应：

- 直接承担数据库结构含义
- 混入响应字段

### 6.2 VO

VO 只用于对外响应：

- 字段只保留调用方需要的内容
- 不暴露内部实现字段
- 不暴露敏感字段
- 名称应表达业务语义，而不是数据库语义

### 6.3 Model

Model 只用于数据映射：

- 字段与表结构保持一致
- 统一时间字段命名
- 敏感字段避免序列化输出
- 不在 model 中堆叠复杂业务方法

## 7. 命名规范

- 导出结构体、接口、方法：`PascalCase`
- 局部变量、函数参数：`camelCase`
- 构造函数：`NewXxx`
- DTO：`XxxRequest` / `XxxQuery`
- VO：`XxxVO`
- Repository：`XxxRepository`
- Service：`XxxService`
- Handler：`XxxHandler`

命名要求：

- 优先语义明确
- 不使用 `data`、`info`、`manager` 这类低信息名称作为核心类型名
- 同一概念在不同层命名保持一致

## 8. 错误处理规范

- 所有业务错误都应可分类
- 不能把所有失败都打成 `500`
- 错误应在 service 层产生，在接入层映射为响应
- 错误文案应可定位问题，但不要泄露敏感实现细节

至少区分：

- 参数错误
- 未认证
- 无权限
- 资源不存在
- 状态冲突
- 系统内部错误

## 9. 事务与数据访问规范

- 多表写入默认由 service 控制事务
- repository 不自己偷偷开启跨业务事务
- 查询条件必须显式
- 更新字段必须显式，避免误覆盖
- 列表查询必须有稳定排序
- 分页参数、过滤参数、排序参数不得依赖调用方隐式约定

## 10. 依赖注入规范

- 核心依赖通过构造函数注入
- 不使用全局变量承载核心 service 或 repository
- router 或 bootstrap 负责最终装配依赖
- 接口抽象只在确实有替换价值时使用，不为抽象而抽象

## 11. 日志规范

- 记录关键失败路径
- 记录外部依赖调用失败
- 日志应包含能定位请求的最小上下文
- 不记录密码、token、隐私原文

## 12. 测试要求

最低要求：

- 核心 service 有基本单元测试或等价验证
- 关键错误分支有覆盖
- repository 至少能通过真实或受控环境验证主要查询

不要求：

- 为简单 getter / setter 写无意义测试
- 在早期小项目里追求虚高覆盖率

## 13. 提交前自检

提交前至少检查：

- 是否出现跨层直接调用
- 是否出现超长函数未拆分
- 是否出现无用导出类型
- 是否出现未使用代码和调试输出
- 是否出现字段定义与文档不一致
