# Git 提交规范

本文档只描述代码协作和提交规范，不承载业务模块拆分说明。

## 1. 目标

提交规范的目标不是追求形式，而是保证：

- 每次提交都能单独理解
- 回滚某次提交时不会破坏大量无关内容
- 代码审查时能快速看清本次改动意图

## 2. 提交粒度

一个提交应满足：

- 改动目标明确
- 改动范围集中
- 提交说明可以用一句话准确概括

适合单独提交的内容：

- 一个清晰的 bug 修复
- 一个独立的重构
- 一份规范文档更新
- 一个独立界面结构调整
- 一组相关接口封装完善

不适合单独提交的内容：

- 无关的前后端改动混在一起
- 重构、功能、样式、文档一起提交
- 大量格式化改动夹带真实逻辑修改

## 3. 提交格式

推荐格式：

```text
type(scope): summary
```

说明：

- `type`
  - 改动类型
- `scope`
  - 改动范围，可选但推荐写
- `summary`
  - 一句话描述结果

## 4. 常用类型

- `feat`
  - 新增能力
- `fix`
  - 修复问题
- `refactor`
  - 重构但不改变外部行为
- `docs`
  - 文档修改
- `test`
  - 测试相关
- `style`
  - 纯格式或样式调整
- `chore`
  - 杂项维护

## 5. 提交文案要求

提交文案必须：

- 直接描述结果
- 避免空泛词
- 避免把多个动作塞进一句话

推荐：

- `fix(login): handle expired session redirect`
- `refactor(network): split request client and modules`
- `docs(standards): expand frontend code guidelines`

不推荐：

- `update code`
- `fix bugs`
- `change files`

## 6. 提交顺序建议

如果改动较大，建议拆分为多个 commit：

1. 目录或基础设施调整
2. 核心逻辑改动
3. 样式或展示调整
4. 文档与测试补充

这样做的好处是：

- review 更清楚
- 回滚更安全
- 冲突更少

## 7. 提交前检查

提交前至少检查：

- 当前分支是否正确
- 是否混入无关文件
- 调试代码是否清理
- 构建或测试是否已验证
- 文档是否需要同步更新
- 是否只提交了预期文件

## 8. 禁止事项

- 在主分支上长期堆积开发
- 把未完成实验代码直接提交
- 一次提交混入多个无关模块
- 把格式化噪音和真实逻辑修改混成无法审查的 diff

## 9. 文档同步规则

以下情况通常需要同步文档：

- 接口字段变化
- 数据结构变化
- 目录结构变化
- 代码规范变化
- 开发流程变化
