# 前端代码规范

本文档只描述前端工程实现规范，不承载任何具体业务页面或功能模块说明。

## 1. 目标

这份规范用于约束：

- 前端目录结构
- 页面与组件职责
- 状态管理边界
- 接口封装方式
- 路由组织方式
- 样式组织方式
- 命名、文件大小和拆分标准

目标是让前端代码在多人协作时保持结构稳定，不因为页面增多而迅速失控。

## 2. 总体原则

所有前端代码默认遵守以下原则：

- 单一职责
  - 页面负责编排，组件负责复用，store 负责共享状态，api 负责请求
- 先分层，再实现
  - 不在页面里同时堆叠请求、状态、样式和大段业务流程
- 显式数据流
  - 数据从哪里来、在哪里改、改完如何同步，必须一眼能看出
- 可维护优先
  - 优先拆分清晰结构，不追求一文件写完全部逻辑
- 复用有边界
  - 真正重复的结构再抽组件，不做过早抽象

## 3. 推荐目录结构

推荐 `src` 结构如下：

```text
src/
  api/
    modules/
  assets/
  components/
  composables/
  constants/
  layouts/
  pages/
  router/
    guards/
  stores/
  styles/
  types/
  utils/
```

目录职责：

- `api/modules`
  - 按资源或模块封装请求
- `assets`
  - 图片、图标、静态资源
- `components`
  - 可复用组件
- `composables`
  - 可复用状态逻辑和行为逻辑
- `constants`
  - 路由名、固定配置、常量枚举
- `layouts`
  - 页面壳和通用布局
- `pages`
  - 页面级视图
- `router`
  - 路由注册
- `router/guards`
  - 路由守卫
- `stores`
  - 共享状态
- `styles`
  - token、全局样式、reset、工具类
- `types`
  - 类型定义
- `utils`
  - 纯工具函数

## 4. 页面与组件职责

### 4.1 页面职责

页面只负责：

- 组织页面布局
- 触发页面级加载
- 组合业务组件
- 管理少量页面本地状态

页面不应负责：

- 直接封装底层请求细节
- 承担多个可复用区块的内部实现
- 长期持有过多共享状态

### 4.2 组件职责

组件只负责可复用 UI 或局部交互。

组件要求：

- 输入通过 `props`
- 输出通过 `emit`
- 不依赖外部页面隐式上下文
- 不直接修改不属于自己的全局状态

组件应拆分的信号：

- 同一结构在两个以上页面重复出现
- 单组件同时负责列表、表单、弹窗、接口请求
- 单组件超过 200 行且阅读成本明显升高

## 5. 单一职责规则

以下规则必须执行：

- 页面文件只解决一个主任务
- store 只管理共享状态和共享动作
- composable 只封装一类交互逻辑或状态逻辑
- api 模块只封装同一资源的请求
- util 只放纯函数，不放页面状态逻辑

如果一个文件同时出现下面三类以上内容，应优先拆分：

- 接口请求
- 大量模板结构
- 状态同步
- 表单校验
- 弹窗逻辑
- 表格或列表渲染

## 6. 状态管理规范

状态按作用域划分：

- 页面本地状态
  - 只在当前页面短期使用
- 组件内部状态
  - 只服务当前组件
- 共享状态
  - 多页面或多组件共同依赖，进入 store

store 只负责：

- 共享状态持有
- 共享数据加载
- 共享状态更新

store 不应负责：

- 大量模板展示判断
- 页面弹窗开关的全部细节
- 与单一页面强绑定的短期临时状态

## 7. 接口封装规范

- 所有 HTTP 请求统一放在 `api` 层
- 页面和组件不直接写裸请求
- 一个资源对应一个 `api/modules/*.ts`
- 请求参数和响应结果都应声明类型
- 接口错误处理应有统一入口或统一风格

禁止：

- 在多个页面复制同一请求逻辑
- 在组件内部直接写一组无法复用的请求封装
- 在请求返回后立刻到处手动改同一份共享状态

## 8. 类型规范

- 接口入参、出参、共享数据结构必须声明类型
- 不在页面内部堆大段匿名对象类型
- 枚举状态和值类型尽量单独集中定义
- 可选字段必须显式标记，不靠猜测使用

## 9. 路由规范

- 路由按模块拆分，不在单文件中堆所有页面
- 路由守卫统一放在 `router/guards`
- 路由 meta 只放轻量配置，不放复杂逻辑
- 权限控制通过守卫或上层状态判断完成，不散落在每个页面里重复实现

## 10. 样式规范

- 设计 token 统一放在 `styles`
- 页面样式优先局部化
- 全局样式只放 reset、基础排版、工具类、主题变量
- 不在多个页面重复定义同一套颜色、间距、阴影规则

禁止：

- 每个页面自带一套独立 token
- 在组件内写大量无法复用的硬编码样式值
- 把业务逻辑判断直接写进复杂样式表达式里

## 11. 命名规范

- 组件：`PascalCase.vue`
- 页面：`PascalCase.vue`
- composable：`useXxx.ts`
- store：`xxx.store.ts` 或项目统一约定格式
- API 文件：按资源命名，如 `user.ts`、`course.ts`
- 类型文件：按资源命名，如 `user.ts`、`course.ts`

命名要求：

- 名称必须表达职责
- 避免 `data`、`list`、`temp` 这种低信息文件名作为核心模块名
- 同一概念在页面、store、api、types 中保持一致命名

## 12. 文件大小与拆分建议

建议控制范围：

- 页面文件：300 行以内
- 组件文件：200 行以内
- store 文件：200 行以内
- composable 文件：150 行以内

超出后优先考虑：

- 拆子组件
- 拆 composable
- 拆表单区块
- 拆请求封装
- 拆纯函数工具

## 13. 交互与异常处理规范

- 所有异步操作要有 loading 状态
- 所有失败路径要有明确反馈
- 空态、加载态、失败态要显式处理
- 表单提交前要有基本校验
- 弹窗关闭后应能恢复到可预期状态

## 14. 禁止事项

- 页面直接写底层请求实现
- 组件偷偷改全局状态且没有清晰入口
- 把权限判断完全散落到按钮显隐中
- 为避免拆分而把所有逻辑堆在单页面中
- 引入明显超出项目体量的复杂架构

## 15. 提交前自检

提交前至少检查：

- 页面是否过长且未拆分
- 请求是否都从 `api` 层进入
- 共享状态是否被错误地放在页面本地
- 是否存在重复组件或重复逻辑未提取
- 是否存在未使用状态、未使用 props、未使用样式
