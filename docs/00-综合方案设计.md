# 课程平台综合方案设计

## 1. 项目目标

本项目面向“课程管理 + 资料管理 + 课程问答 Agent”场景，核心目标如下：

1. 支持用户注册、登录、查看个人信息。
2. 支持课程的创建、成员管理、角色授权、课程内容维护。
3. 支持按课程隔离资料空间，完成资料上传、目录展示、内容预览、权限控制。
4. 支持为每门课程绑定一个课程 Agent，学生可基于课程资料进行提问，Agent 仅访问本课程授权范围内的资料。

本方案只关注业务功能、接口设计、权限规则与数据结构，不绑定具体技术栈。

## 2. 核心角色

系统内包含普通用户身份，以及课程内角色体系。

### 2.1 课程内角色

1. 创建者 `owner`
   - 创建课程。
   - 拥有课程最高权限。
   - 可删除课程。
   - 可转移创建者身份。
2. 教师 `teacher`
   - 可编辑课程信息。
   - 可管理课程资料。
   - 可配置课程 Agent。
   - 可查看课程内学生列表。
3. 学生 `student`
   - 可查看课程信息。
   - 可查看和下载课程资料。
   - 可向课程 Agent 提问。

## 3. 核心业务对象

1. 用户：平台登录主体。
2. 课程：课程基本实体。
3. 课程成员：用户与课程之间的关系及角色。
4. 资料空间：每门课程独立文件空间的抽象。
5. 资料节点：课程文件夹和文件的树形结构。
6. Agent：课程维度的智能问答配置。
7. Agent 对话：学生与课程 Agent 的问答记录。

## 4. 权限总览

| 功能 | 未登录 | 学生 | 教师 | 创建者 |
| --- | --- | --- | --- | --- |
| 注册/登录 | 支持 | - | - | - |
| 查看个人信息 | 否 | 是 | 是 | 是 |
| 查看课程详情 | 否 | 是 | 是 | 是 |
| 创建课程 | 否 | 是 | 是 | 是 |
| 编辑课程信息 | 否 | 否 | 是 | 是 |
| 删除课程 | 否 | 否 | 否 | 是 |
| 查看课程成员 | 否 | 限本人可见范围 | 是 | 是 |
| 管理课程成员 | 否 | 否 | 是 | 是 |
| 浏览资料树 | 否 | 是 | 是 | 是 |
| 上传资料 | 否 | 否 | 是 | 是 |
| 删除资料 | 否 | 否 | 是 | 是 |
| 配置 Agent | 否 | 否 | 是 | 是 |
| 向 Agent 提问 | 否 | 是 | 是 | 是 |
| 查看全部对话记录 | 否 | 否 | 是 | 是 |

说明：

1. 学生是否可查看完整成员列表，可按业务需要控制。默认建议学生只能看到教师列表和自己的成员信息。
2. 教师可删除资料，但不能删除课程。
3. 创建者默认同时具备教师能力。

## 5. 系统模块划分

1. 用户模块
   - 登录、注册、身份识别、个人资料查看。
2. 课程模块
   - 课程创建、课程详情、成员与角色管理、课程信息编辑、课程删除。
3. 资料模块
   - 课程文件空间、文件夹树、文件上传、文件预览、逻辑删除、权限控制。
4. Agent 模块
   - 课程 Agent 配置、资料检索范围控制、问答接口、对话记录、引用来源返回。

## 6. 数据库设计原则

1. 用户、课程、课程成员解耦，便于扩展课程内多角色体系。
2. 文件系统采用“数据库记录元信息 + 本地文件落盘”的方式。
3. 文件树通过 `parent_id` 维护层级关系，不直接依赖真实磁盘结构。
4. Agent 对课程和资料建立显式绑定，避免跨课程误访问。
5. 所有关键业务表建议包含创建时间、更新时间、创建人、状态字段。

## 7. 数据库表结构总览

本方案建议的核心表如下：

1. `users`：用户表
2. `courses`：课程表
3. `course_members`：课程成员表
4. `course_storage_spaces`：课程资料空间表
5. `course_material_nodes`：课程资料节点表
6. `course_material_versions`：资料版本表
7. `course_agents`：课程 Agent 配置表
8. `agent_conversations`：Agent 会话表
9. `agent_messages`：Agent 消息表
10. `agent_message_sources`：Agent 回答引用资料表

登录态不落数据库表，建议统一存储在 Redis 中。

## 8. 参考 SQL

以下 SQL 仅作为逻辑结构示例，不绑定数据库类型细节。

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    username VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    phone VARCHAR(32),
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

CREATE TABLE courses (
    id BIGINT PRIMARY KEY,
    course_code VARCHAR(64) NOT NULL UNIQUE,
    course_name VARCHAR(128) NOT NULL,
    course_description TEXT,
    owner_user_id BIGINT NOT NULL,
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (owner_user_id) REFERENCES users(id)
);

CREATE TABLE course_members (
    id BIGINT PRIMARY KEY,
    course_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    role VARCHAR(16) NOT NULL,
    join_status VARCHAR(16) NOT NULL DEFAULT 'active',
    joined_at DATETIME NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    UNIQUE (course_id, user_id),
    FOREIGN KEY (course_id) REFERENCES courses(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE course_storage_spaces (
    id BIGINT PRIMARY KEY,
    course_id BIGINT NOT NULL UNIQUE,
    root_path VARCHAR(255) NOT NULL,
    quota_bytes BIGINT NOT NULL DEFAULT 0,
    used_bytes BIGINT NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (course_id) REFERENCES courses(id)
);

CREATE TABLE course_material_nodes (
    id BIGINT PRIMARY KEY,
    course_id BIGINT NOT NULL,
    space_id BIGINT NOT NULL,
    parent_id BIGINT,
    node_type VARCHAR(16) NOT NULL,
    node_name VARCHAR(255) NOT NULL,
    file_ext VARCHAR(32),
    storage_path VARCHAR(512),
    mime_type VARCHAR(128),
    file_size BIGINT NOT NULL DEFAULT 0,
    latest_version_no INT NOT NULL DEFAULT 1,
    sort_index INT NOT NULL DEFAULT 0,
    is_deleted TINYINT NOT NULL DEFAULT 0,
    created_by BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (course_id) REFERENCES courses(id),
    FOREIGN KEY (space_id) REFERENCES course_storage_spaces(id),
    FOREIGN KEY (parent_id) REFERENCES course_material_nodes(id),
    FOREIGN KEY (created_by) REFERENCES users(id)
);

CREATE TABLE course_material_versions (
    id BIGINT PRIMARY KEY,
    material_node_id BIGINT NOT NULL,
    version_no INT NOT NULL,
    storage_path VARCHAR(512) NOT NULL,
    file_size BIGINT NOT NULL,
    mime_type VARCHAR(128),
    upload_user_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    UNIQUE (material_node_id, version_no),
    FOREIGN KEY (material_node_id) REFERENCES course_material_nodes(id),
    FOREIGN KEY (upload_user_id) REFERENCES users(id)
);

CREATE TABLE course_agents (
    id BIGINT PRIMARY KEY,
    course_id BIGINT NOT NULL UNIQUE,
    agent_name VARCHAR(128) NOT NULL,
    prompt_template TEXT,
    status VARCHAR(16) NOT NULL DEFAULT 'enabled',
    retrieval_scope VARCHAR(16) NOT NULL DEFAULT 'course_all',
    created_by BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (course_id) REFERENCES courses(id),
    FOREIGN KEY (created_by) REFERENCES users(id)
);

CREATE TABLE agent_conversations (
    id BIGINT PRIMARY KEY,
    course_id BIGINT NOT NULL,
    agent_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    conversation_title VARCHAR(255),
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (course_id) REFERENCES courses(id),
    FOREIGN KEY (agent_id) REFERENCES course_agents(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE agent_messages (
    id BIGINT PRIMARY KEY,
    conversation_id BIGINT NOT NULL,
    sender_type VARCHAR(16) NOT NULL,
    message_content TEXT NOT NULL,
    token_usage INT,
    created_at DATETIME NOT NULL,
    FOREIGN KEY (conversation_id) REFERENCES agent_conversations(id)
);

CREATE TABLE agent_message_sources (
    id BIGINT PRIMARY KEY,
    message_id BIGINT NOT NULL,
    material_node_id BIGINT NOT NULL,
    material_version_id BIGINT,
    snippet_text TEXT,
    created_at DATETIME NOT NULL,
    FOREIGN KEY (message_id) REFERENCES agent_messages(id),
    FOREIGN KEY (material_node_id) REFERENCES course_material_nodes(id),
    FOREIGN KEY (material_version_id) REFERENCES course_material_versions(id)
);
```

## 9. 关键业务规则

### 9.1 用户与课程

1. 用户创建课程时，自动成为课程创建者。
2. 课程创建者必须同时出现在课程成员表中，角色为 `owner`。
3. 一个用户在同一课程中只能有一条有效成员关系。
4. 用户登录态统一保存在 Redis，数据库不维护会话表。

### 9.2 课程与资料空间

1. 每门课程只对应一个资料空间。
2. 创建课程时自动初始化课程资料空间。
3. 资料空间对应一个本地根目录，例如 `/storage/course/{courseId}/`。

### 9.3 文件树

1. 逻辑目录结构以数据库为准。
2. 本地磁盘用于存储真实文件内容，不作为权限判断依据。
3. 删除资料优先采用逻辑删除，避免误删与便于审计。

### 9.4 Agent 检索范围

1. Agent 只能访问所属课程下未删除且已授权的资料。
2. 若后续需要细粒度控制，可增加“资料是否可用于 Agent 检索”的标记字段。
3. Agent 回答应返回引用来源，至少包括文件名、节点 ID、片段摘要。

## 10. 接口设计总原则

1. 接口按模块分组，统一使用身份认证。
2. 只要涉及课程资源，必须先校验是否属于该课程成员。
3. 涉及写操作，必须校验课程角色。
4. 接口返回中应包含明确的权限失败原因，如“不是课程成员”或“不是教师/创建者”。
5. 文件下载、预览、Agent 检索都必须基于课程范围进行隔离。

## 11. 统一返回格式

所有业务接口建议使用统一 JSON 返回结构。

### 11.1 成功返回格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

1. `code`：业务状态码，`0` 表示成功，非 `0` 表示失败。
2. `message`：简短说明，成功时固定为 `success`，失败时返回可读错误信息。
3. `data`：实际业务数据，查询接口返回对象或数组，删除或退出类接口可返回 `null`。

### 11.2 失败返回格式

```json
{
  "code": 40001,
  "message": "用户名已存在",
  "data": null
}
```

### 11.3 建议错误码范围

1. `0`：成功
2. `40000-40999`：通用请求错误，例如参数错误、重复提交
3. `41000-41999`：认证错误，例如未登录、登录失效
4. `42000-42999`：权限错误，例如不是课程成员、不是教师、不是创建者
5. `43000-43999`：资源错误，例如课程不存在、文件不存在、会话不存在
6. `44000-44999`：业务状态错误，例如课程已删除、Agent 已禁用
7. `50000-50999`：系统内部错误

### 11.4 常用错误码示例

| 错误码 | 说明 |
| --- | --- |
| 0 | 成功 |
| 40001 | 参数校验失败 |
| 40002 | 用户名已存在 |
| 41001 | 未登录 |
| 41002 | 登录态已失效 |
| 42001 | 无课程访问权限 |
| 42002 | 无教师权限 |
| 42003 | 无创建者权限 |
| 43001 | 用户不存在 |
| 43002 | 课程不存在 |
| 43003 | 资料不存在 |
| 43004 | 会话不存在 |
| 44001 | 课程已删除 |
| 44002 | Agent 已禁用 |
| 50000 | 系统异常 |

## 13. 实现重点建议

1. 先完成用户、课程、资料三大基础模块，再接入 Agent。
2. Agent 模块必须依赖课程成员身份校验和资料元数据管理。
3. 文件预览与 Agent 检索都不应直接操作原始目录遍历，而应先查数据库节点。
4. 如果后续增加作业、考试、通知等模块，课程与成员体系可以直接复用。
