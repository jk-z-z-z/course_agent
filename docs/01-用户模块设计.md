# 用户模块设计

## 1. 模块目标

用户模块负责提供平台基础身份能力，是所有课程与资料访问的前置模块。目标如下：

1. 支持用户注册。
2. 支持用户登录与退出登录。
3. 支持查看当前登录用户个人信息。
4. 为课程、资料、Agent 模块提供统一身份标识。

## 2. 模块边界

用户模块负责：

1. 账号注册。
2. 账号认证。
3. 登录态维护。
4. 个人信息查询。

用户模块不负责：

1. 课程角色判断。
2. 课程成员关系维护。
3. 文件权限判断。
4. Agent 业务处理。

## 3. 数据库表结构

### 3.1 用户表 `users`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT | 主键 |
| username | VARCHAR(64) | 登录账号，唯一 |
| password_hash | VARCHAR(255) | 密码摘要 |
| phone | VARCHAR(32) | 手机号 |
| status | VARCHAR(16) | 用户状态，`active`/`disabled` |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 3.2 登录态存储

登录态不使用数据库表，统一保存在 Redis。

建议存储内容：

1. `token -> userId`
2. `token -> 过期时间`
3. 可选扩展当前设备信息、最近访问时间

## 4. 参考 SQL

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    username VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    phone VARCHAR(32),
    status VARCHAR(16) NOT NULL DEFAULT 'active',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

## 5. 功能说明

### 5.1 注册

用户通过用户名和密码完成注册，系统创建用户记录。

建议规则：

1. 用户名唯一。
2. 密码只保存摘要，不保存明文。
3. 可选填写手机号。

### 5.2 登录

用户使用用户名和密码登录，系统校验通过后生成登录态。

建议规则：

1. 校验用户是否存在。
2. 校验密码是否正确。
3. 校验账号是否被禁用。
4. 登录成功后生成 token，并写入 Redis。

### 5.3 退出登录

用户主动注销当前登录态，系统使对应会话失效。

### 5.4 查看个人信息

已登录用户可查看自己的基础资料。

## 6. 接口设计

本模块所有接口响应都应遵循统一返回格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 6.1 注册

接口：`POST /api/users/register`

功能：

1. 创建平台账号。

请求参数建议：

```json
{
  "username": "zhangsan",
  "password": "******",
  "phone": "13800000000"
}
```

响应内容建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "userId": 1001,
    "username": "zhangsan"
  }
}
```

实现要点：

1. 校验用户名是否重复。
2. 密码进行摘要处理。
3. 对手机号格式做基础校验。

接口范围：

1. 仅负责创建账号。
2. 不负责自动加入任何课程。

权限规则：

1. 未登录用户可调用。
2. 已登录用户通常不需要重复调用。

### 6.2 登录

接口：`POST /api/users/login`

功能：

1. 认证用户身份。
2. 生成登录态。

请求参数建议：

```json
{
  "username": "zhangsan",
  "password": "******"
}
```

响应内容建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "session-token",
    "expiredAt": "2026-07-31T23:59:59",
    "user": {
      "id": 1001,
      "username": "zhangsan",
      "phone": "13800000000"
    }
  }
}
```

实现要点：

1. 根据用户名查找用户。
2. 比对密码摘要。
3. 生成 token 并写入 Redis。

接口范围：

1. 只负责登录认证。
2. 不返回课程列表等扩展业务数据。

权限规则：

1. 未登录用户可调用。

### 6.3 退出登录

接口：`POST /api/users/logout`

功能：

1. 使当前登录态失效。

响应内容建议：

```json
{
  "code": 0,
  "message": "success",
  "data": null
}
```

实现要点：

1. 根据当前 token 删除 Redis 中的登录态。

接口范围：

1. 仅处理当前用户会话。

权限规则：

1. 仅登录用户可调用。

### 6.4 查看当前用户信息

接口：`GET /api/users/me`

功能：

1. 获取当前登录用户资料。

响应内容建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1001,
    "username": "zhangsan",
    "phone": "13800000000"
  }
}
```

### 6.5 错误返回示例

以登录失败为例：

```json
{
  "code": 41002,
  "message": "用户名或密码错误",
  "data": null
}
```

实现要点：

1. 从登录态中识别当前用户。
2. 返回脱敏后的必要字段。

接口范围：

1. 仅返回当前用户自身资料。
2. 不提供任意用户详情查询能力。

权限规则：

1. 仅登录用户可调用。

## 7. 权限规则

1. 注册、登录属于匿名接口。
2. 个人信息接口依赖登录态。
3. 用户模块只判断“是否登录”，不判断课程角色。

## 8. 实现注意事项

1. 用户状态为禁用时，不允许登录。
2. 用户删除不建议物理删除，应保留以维护课程历史数据完整性。
3. 所有其他业务模块都应通过用户 ID 关联，不直接依赖用户名。
4. 登录态建议使用 Redis 过期机制管理，不在关系型数据库中单独建表。
